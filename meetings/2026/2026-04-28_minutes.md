### Devices and Sensors WG 2026 Charter Review

* Simone Onofri: The Devices and Sensors WG charter raises long-running questions about web pages connecting to device capabilities such as USB, NFC, Bluetooth, sensors, and other device components.

* Simone Onofri: The group should review the charter, the W3C Strategy issue, and the background paper linked from the agenda.

* Simone Onofri: The intended goal of the Devices and Sensors work is to provide secure and privacy-preserving access to device capabilities, but the security and privacy discussion is still significant.

* Simone Onofri: One concrete concern is that device APIs may expose static or persistent identifiers. For example, a website interacting with a connected phone may be able to recognize the same device or user across interactions.

* Simone Onofri: These APIs therefore raise threat modeling questions around use cases, exposed capabilities, threats, mitigations, and acceptable residual risk.

* Simone Onofri: The relevant device capability areas include sensors, accelerometer, gyroscope, and other hardware interfaces.

* Simone Onofri: The discussion needs to balance the business and developer need for better web application integration with devices against the security and privacy consequences of exposing those capabilities to web origins.

* Simone Onofri: The paper mentioned in the agenda appears useful because it discusses techniques for using external device APIs in ways that can cross or bypass ordinary browser sandbox assumptions.

* Simone Onofri: The security and privacy review for the charter still appears to be needed, and SING should participate in the discussion.

* Simone Onofri: The preferred next step may be to ask the Devices and Sensors WG to draft a threat model that identifies the relevant use cases, threats, and mitigations before the residual risks are evaluated.

* Simone Onofri: The reference to TAG in the discussion means the Technical Architecture Group, not an HTML tag; the capabilities under discussion are exposed through JavaScript APIs.

### Threat Model for the Web DFD

* Giovanni Corti: The current Editor's Draft of Threat Model for the Web includes the Web Security Model and a new Data Flow Diagram.

* Giovanni Corti: The Data Flow Diagram is the most recent update to the draft and shows how the web browser interacts with other entities.

* Giovanni Corti: This work is proceeding in parallel with the browser vertical harmonized standard ETSI EN 304-617, and the goal is to make the Threat Model for the Web referenceable from that work.

* Giovanni Corti: Feedback on the Data Flow Diagram is welcome, and the group should consider opening a call for consensus to publish the draft as a First Public Note Draft.

* Giovanni Corti: The goal of the diagram is to identify relevant trust boundaries, data flows, and eventually threats, using a generic and vendor-agnostic browser architecture.

* Giovanni Corti: The diagram includes the main privileged browser process, the render or web content process, the network process, GPU/media/helper processes, sandboxed internal content execution processes, web browser extensions, the host environment, and rendered web applications.

* Giovanni Corti: The diagram also includes examples of untrusted web content, including different trusted and malicious origin and iframe combinations.

* Simone Onofri: The diagram is useful for the Devices and Sensors discussion because it already helps show trust-boundary crossings for capabilities such as geolocation, microphone, and camera.

* Simone Onofri: The diagram may need to be extended for other risky capabilities, such as USB and related device interfaces.

* Simone Onofri: Publication is useful because the document could then be referenced by the European browser standardization work.

* Simone Onofri: As one of the editors, Simone expressed support for publication but emphasized that the group should proceed by consensus.

* Simone Onofri: The group was asked whether there were objections to publishing the Threat Model for the Web as a First Public Note Draft.

* Nishant Jain: The diagram looks good, although more time is needed to study it before providing detailed comments.

* Simone Onofri: No objections were heard, so the next step is to send a call for consensus.
