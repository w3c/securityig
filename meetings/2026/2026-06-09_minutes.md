### WebMCP Early Review and Threat Modeling Scope

* Simone Onofri: The WebMCP explainer is the README, and it includes useful diagrams. The work also has a substantial security and privacy considerations document.

* Simone Onofri: I think the first step is to decide when SING should meet to analyze WebMCP, identify volunteers, and determine whether to use the next SING meeting to begin the discussion.

* Simone Onofri: Nishant Jain has volunteered for the review, and Tommaso Innocenti also expressed interest in contributing.

* Tommaso Innocenti: I asked the Google participants what they expected from the review and whether there were particular areas that should be in scope.

* Johann Hofmann: The WebMCP group is open to a threat modeling exercise. I do not think that a formal threat model has already been produced for the work.

* Johann Hofmann: AI security on the Web is still a very new field. Existing MCP integrations are often based on a trust model where a developer explicitly adds an MCP server to a coding agent and accepts some level of trust in it.

* Johann Hofmann: I do not think that trust-based model translates cleanly to the open Web. Open MCP discovery, and the possibility that agents may encounter and use tools from untrusted or harmful sources, is a major challenge.

* Johann Hofmann: In addition to AI-specific concerns, I see traditional Web platform security questions, including exposing communication channels to websites, allowing iframes to declare tools to parent contexts, and considering whether tools may be called by AI agents or by deterministic JavaScript code.

* Tom Jones: My major concern is the loose use of the term “context”. I suggested that context should be treated as a first-class object, with a defined lifetime and a clear point at which the context is destroyed.

* Tom Jones: Context is user data, so I think we need to understand where it comes from, how long it persists, and what happens when a user switches to a new topic or starts a new chat.

* Johann Hofmann: Tracking provenance of context inside current LLM-based systems is a fundamental challenge. Once data is exposed to the model, it is difficult to trace how that data is represented or used.

* Johann Hofmann: One important design question for me is whether some information, such as sensitive tokens or passwords, can be kept from being exposed to the model at all.

* Johann Hofmann: I think some of these questions are currently more in the hands of implementers than in the Web platform layer.

* Tom Jones: I think treating context handling only as an implementation question is problematic; it should also be understood as a policy question.

* Johann Hofmann: Starting a new chat may create a fresh context for the model or inference system, but surrounding or ambient context may still remain, such as the page the agent is viewing or other state provided by the agentic harness.

* Tom Jones: I think the terminology around LLMs, inference engines, MCP hosts, websites, and pages needs to be clarified because different communities may use the same words differently.

* Johann Hofmann: In this discussion, when I say “model”, I generally mean the system running the model in the backend, often together with an agentic harness that provides tools, system instructions, and surrounding context.

* Tommaso Innocenti: I see the review as a good opportunity to clarify terminology, scope, and expectations before attempting a deeper threat model.

* Tom Jones: I suggested that the group may need a broader security paper on AI on the Web. I am concerned that breaking the problem into small pieces may make it harder to understand the whole threat model.

* Johann Hofmann: I agree that a broader paper on AI on the Web could be useful, but I do not think it would necessarily be actionable for the WebMCP team. The immediate request is specifically about WebMCP.

* Simone Onofri: I think we should aim to produce something useful for the WebMCP specification, not only a theoretical discussion about AI.

* Simone Onofri: The practical objective is to help the WebMCP group understand security and privacy questions for the specification.

* Simone Onofri: I think it would be useful to read the documentation, try the technology where possible, and understand how the current implementation can be exercised.

* Johann Hofmann: WebMCP is currently available through a Chrome Origin Trial and is not fully launched.

* Johann Hofmann: WebMCP is not currently supported by Chrome's built-in browser agent. Gemini in Chrome does not yet support it.

* Johann Hofmann: It should be possible to experiment with WebMCP using flags, localhost, or an extension that can use the API and prompt an agent with an API key.

* Simone Onofri: My working understanding is that WebMCP tries to solve the problem that, today, an agent interacting with a website often relies on screenshots, reading the DOM, and clicking. WebMCP would instead let the website communicate capabilities or tools to the agent in a more structured way.

* Johann Hofmann: I think that summary is a good way to describe the intended problem space.

* Simone Onofri: I think the review should compare the world without WebMCP and the world with WebMCP, so that we can understand the security and privacy trade-offs.

* Johann Hofmann: WebMCP and MCP have different concepts. WebMCP borrows ideas around tool descriptions, but the tools are executed differently.

* Johann Hofmann: In the current WebMCP model, the agent loads and renders the website in the browser. The site then registers WebMCP tools, the agent sees which tools are available, and the agent may decide to call them.

* Johann Hofmann: The tools are not executed by calling a separate MCP server. They are JavaScript functions on the web page, with the normal JavaScript capabilities of that page.

* Tom Jones: I think this distinction is important, because otherwise it is unclear whether the agent is interacting with an MCP service, a web service, an HTTP endpoint, or JavaScript running in the page.

* Tommaso Innocenti: The goal of this meeting was not to threat model immediately, but to clarify the scope and expectations for a later threat modeling exercise, possibly on 16 June 2026.

* Julia Pagnucco: I can attend the follow-up session and can prepare an overview of what the team has been thinking about.
