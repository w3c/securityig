### TAG Question: Compression Dictionaries, CORP, and Cross-Origin Boundaries

* Simone Onofri: The TAG asked SING to look at a question that may affect security, although this is not a full security review.

* Simone Onofri: The topic concerns a mechanism standardized in the IETF as an HTTP-related feature, but it interacts with Fetch and therefore sits across IETF, WHATWG, and W3C-related web security boundaries.

* Simone Onofri: I tried to model the issue using an AI-assisted approach similar to the one Nishant used, by providing the Threat Modeling Guide, the Security and Privacy Questionnaire, and the browser fingerprinting guidance as context.

* Simone Onofri: The feature uses dictionaries for compression. The question is whether dictionaries should be allowed in cross-origin situations, which would relax part of the usual same-origin and partitioning assumptions.

* Simone Onofri: RFC 9842 has useful security and privacy considerations, and the main concern appears to be that the feature could become a side channel for tracking or fingerprinting.

* Simone Onofri: The web platform already distinguishes CORS and CORP. CORS governs cross-origin access, while CORP is an HTTP response header through which the resource owner can restrict whether a resource may be used in another origin’s context.

* Simone Onofri: The Fetch model may allow the browser to fetch a resource while still preventing it from being exposed to the rendering context. The TAG question asks whether an exception should be created so that these dictionaries can be loaded despite CORP restrictions.

* Simone Onofri: My current interpretation is that the issue may be more privacy-sensitive than purely security-sensitive, although it also concerns the integrity of the Fetch security boundary.

* Simone Onofri: The privacy section in the RFC is shorter than the security section, but it may be the stronger and more important part of the analysis.

* Simone Onofri: A lot of the responsibility appears to be transferred to the user agent, and this delegation should be looked at carefully.

* Simone Onofri: I will probably ask privacy reviewers for comments, because the issue raises questions about tracking, partitioning, and cross-origin data use.

* Simone Onofri: I have more questions than answers at this stage, and volunteers are welcome to look at the issue together.

* Tommaso Innocenti: The RFC appears to have been published recently and should be treated as an official RFC rather than an early draft.

* Tommaso Innocenti: It is interesting, but also another example of the number of HTTP headers and cross-layer mechanisms that the group has to reason about.

* Tom Jones: What is the point of the dictionary, and could it be used as input to an AI system?

* Simone Onofri: The MDN example shows the dictionary being used for lossless compression and performance. I do not know whether it could be used by AI, but that does not appear to be the design purpose.

### WebMCP Threat Modeling Session

* Tommaso Innocenti: The next topic is the planned WebMCP threat modeling session.

* Tommaso Innocenti: WebMCP, or Web Context Protocol, is a new W3C-related effort intended to let AI agents interact with web pages through a structured API.

* Tommaso Innocenti: WebMCP may introduce threat vectors that go beyond traditional web development, including malicious agents, tool poisoning, and prompt injection attacks.

* Tommaso Innocenti: A first security considerations section has already been drafted with input from Google and Microsoft, but it needs deeper review and discussion with the people joining the threat modeling session.

* Tommaso Innocenti: The Threat Modeling Community Group session is expected on 7 April.

* Simone Onofri: The WebMCP group invited us to an initial call on 2 April at 15:00 UTC to present the threat modeling approach and set the stage for the later session.

* Simone Onofri: The request for the initial call should be to bring diagrams, existing threats, and any already identified mitigations, so that the Threat Modeling Community Group can either elicit more threats or discuss remediation for known ones.

* Simone Onofri: The upcoming session may need two hours, but even then it may only be possible to address one perspective properly.

* Tom Jones: I have very strong concerns about the direction of the work and would likely be too controversial to be helpful in the WebMCP call.

* Joe Andrieu: I share some of Tom’s concerns, but I will attend because the WebMCP session is also an opportunity to improve the process for doing a quick two-hour threat model.

* Joe Andrieu: The group learned useful lessons from the DID Resolution threat modeling work, including what did and did not work in a compressed threat modeling format.

* Joe Andrieu: I am concerned about the broader tendency to put AI everywhere, and WebMCP seems to be part of that trend.

* Tom Jones: My main concern is privacy. If WebMCP allows a random website to query user data, then it could create a serious privacy problem.

* Tom Jones: The question is not what the technology is intended to do, but what an attacker will use it for, including exfiltrating the user’s entire context.

* Simone Onofri: There appear to be at least two different perspectives to model: agents abusing websites, and websites abusing agents that visit them.

* Simone Onofri: Another relevant angle is whether WebMCP provides a faster way for an agent to understand how to interact with a website, and therefore also a faster way to abuse or compromise that website.

* Simone Onofri: Related mechanisms already appear in the ecosystem, for example through APIs that let agents or crawlers interact with websites, so the same type of threat model may apply beyond WebMCP itself.

* Tommaso Innocenti: One possible framing is whether the agent already has information about the user and whether it may share that information with websites through WebMCP.

* Tom Jones: A more specific concern is that a random website visited by the user could cause code on the user’s computer to interact with another website as that user.

* Tom Jones: If WebMCP is exposed through the DOM or a JavaScript API, then the threat model should consider whether any website could use the user’s context to interact with other sites.

* Joe Andrieu: If WebMCP invoked by a website can contact another server from the user’s computer, that should be considered in the threat model and may require sandboxing.

* Tom Jones: Even if that is not the intended use, it should still be explicitly included in the threat model.

* Simone Onofri: This is exactly the kind of concern that threat modeling should surface. The role of this group is to identify problems clearly; the specification authors then need to decide how to mitigate them.

* Simone Onofri: The fact that security and privacy people identify problems should not be seen as negativity, but as the expected function of the process.

* Tommaso Innocenti: The initial meeting is planned for 2 April, and the Threat Modeling Community Group session for WebMCP is planned for 7 April.

* Simone Onofri: I expect that John from Google, who is competent on security and privacy, may be involved in the security and privacy text and is likely open to hearing concerns from the group.

### Discussion on Using AI in Threat Modeling

* Tommaso Innocenti: The next topic is the use of AI in threat modeling.

* Simone Onofri: This is a discussion we have also been having in the security team and inside W3C, to understand how AI may affect our work.

* Simone Onofri: After the DID-related threat modeling work, we saw examples of AI-generated material that looked professional but included conceptual problems.

* Simone Onofri: One problem is when AI-generated material places different kinds of frameworks or taxonomies, such as STRIDE, LINDDUN, and Shostack’s Four Questions, at the same conceptual level.

* Simone Onofri: The broader question is how we should respond when people submit large AI-generated threat models or security sections that then impose a review burden on other people.

* Simone Onofri: This can create a human time tax, because a person still needs to review and correct the output. We have already experienced a similar problem with very long standards material.

* Simone Onofri: At the same time, I am experimenting with AI on the TAG issue to understand whether there is a sane way to use AI in analysis and threat modeling.

* Joe Andrieu: We need to be encouraging, because people are being asked to do additional work and will naturally try to use AI to help.

* Joe Andrieu: There is an open issue in the Threat Modeling Guide about adding a section on AI.

* Joe Andrieu: The problem with simply using an LLM to generate a threat model is that it misses the main value of threat modeling at W3C.

* Joe Andrieu: The most valuable part is having the people developing the specification, who understand the technology and its design decisions, think about the threats they care about and capture that reasoning in a common form.

* Joe Andrieu: In the future, anyone will be able to point an LLM at a specification and get a generic analysis. The unique value is the working group’s domain knowledge, design rationale, and expectations.

* Joe Andrieu: AI can be useful for ideation, exploring options, improving tone, cleaning up vocabulary, and supporting people working in a non-native language.

* Joe Andrieu: We should avoid the pattern where someone sends a specification to an LLM and then calls the generated output a threat model.

* Simone Onofri: A better framing may be to use AI as a Socratic partner: use it to generate candidate weaknesses or questions, then use those outputs as input for discussion by the group.

* Simone Onofri: AI can help someone understand material, as happened when I used the fingerprinting guidance and other context to reason about the RFC, but it cannot substitute human thinking.

* Simone Onofri: The goal remains to make the working group think about threats. If they want to think through those threats by talking with an AI, that can be useful.

* Joe Andrieu: A risk is that people may think they are reading AI output critically while actually rubber-stamping it, especially after several apparently good outputs in a row.

* Tommaso Innocenti: Guidelines would be useful, but the group cannot fully enforce how people use or trust AI.

* Tommaso Innocenti: Blindly trusting a prompt is not a good approach, even when the output often looks useful.

* Joe Andrieu: The Socratic approach is promising: instead of asking AI to do the task, ask how to approach the task, what options exist, or what may be missing.

* Tommaso Innocenti: This means using AI to refine the discussion or guide reasoning, rather than using it to produce the threat model itself.

* Tom Jones: I often use AI by asking it to identify weaknesses. It usually returns a long list.

* Simone Onofri: That can be useful if the group then evaluates which weaknesses are valid and how they should be mitigated.

* Tom Jones: The problem is that AI often finds too many possible issues, so people still have to select the useful ones.

* Joe Andrieu: That curation is exactly why the specification authors need to be involved, because they know the domain and the design rationale.

* Tom Jones: There is also a persistent tension: specification authors tend to look for the good things, while threat modelers look for the bad things.

* Joe Andrieu: That tension is unavoidable, but it is part of the value of the exercise.

* Simone Onofri: The Threat Modeling Community Group may be a place to experiment with AI-assisted threat modeling, possibly during the WebMCP work.

* Joe Andrieu: That could be interesting, but it could also go wrong quickly and waste the session if not controlled.

* Tom Jones: AI can produce so much material that just going through it may take too long.

* Tom Jones: AI is also useful for attack analysis, but that is usually based on existing attacks. STRIDE is more useful when looking for future or structurally possible attacks.

* Simone Onofri: This reinforces the point that AI can be a good companion, but not a substitute for human threat modeling.
