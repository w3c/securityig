### Tab-isolated Token Protocol

* Tommaso Innocenti: The next topic is **post-injection XSS mitigation** using a **tab-isolated token protocol**.

* Tommaso Innocenti: Harsh will present his proposal. The floor is yours.

* Harsh Singhal: Thank you for the invitation. This is my first time presenting to this group.

* Harsh Singhal: The tab-isolated token protocol was developed to mitigate **cross-site scripting attacks after injection**.

* Harsh Singhal: Existing mechanisms such as CSP and sanitization focus on prevention. However, once a malicious script is injected, the backend cannot distinguish between legitimate application code and injected code.

* Harsh Singhal: The goal is to ensure that even someone who knows the application code cannot retrieve the security token through an injected script.

* Harsh Singhal: The proposal introduces a **new cookie attribute called `TabOnly`**.

* Harsh Singhal: A tab-only cookie exists only within a single browser tab. If the tab is refreshed or closed, the cookie disappears.

* Harsh Singhal: This differs from session cookies, which remain until the browser is closed.

* Harsh Singhal: Using this attribute, we can implement a token mechanism similar to CSRF tokens but aimed at **XSS mitigation**.

* Harsh Singhal: The client generates a **public-private key pair**.

* Harsh Singhal: The public key is sent to the backend server.

* Harsh Singhal: The backend generates:

  * a **token**
  * a **tab ID**

* Harsh Singhal: The **tab ID** is stored in the `TabOnly` cookie.

* Harsh Singhal: The **token** is encrypted with the client’s public key and returned to the client.

* Harsh Singhal: The client decrypts the token using its private key and uses it for subsequent requests.

* Tom Jones: Would some cookies be tab-based while others remain browser-based?

* Harsh Singhal: Yes. Only cookies with the **TabOnly attribute** would be scoped to the tab. Other cookies would behave normally.

* Tommaso Innocenti: How is the public key generated? Is it derived from TLS?

* Harsh Singhal: No. The key pair is generated locally using the browser crypto API (e.g., `crypto.subtle.generateKey`).

* Tommaso Innocenti: Then why generate another key pair if TLS already encrypts the communication?

* Harsh Singhal: TLS protects the transport channel. However, once the response reaches the browser, any JavaScript on the page can read it. An injected script would therefore see the token.

* Harsh Singhal: Encrypting the token with the client’s public key ensures that only the legitimate application code holding the private key can decrypt it.

* Patrick Schaller: Could a simpler hash-based mechanism replace the public-private key pair?

* Harsh Singhal: That approach would still allow a malicious script to observe and reuse the exchanged values. Public-key encryption prevents that because the attacker cannot decrypt the token.

* Tom Jones: It sounds like some aspects relate to application-level security mechanisms that already exist. Have you considered alignment with those?

* Harsh Singhal: My focus is specifically on **post-injection mitigation**. Current mechanisms focus mainly on prevention, but protection after injection is still limited.
